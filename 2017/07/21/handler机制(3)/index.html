Handler机制(3)

enqueueMessage、next

在Handler机制中，分析到Handler的enqueueMessage()方法，它最终的将msg交给了MessageQueue的enqueueMessage()处理。
而Looper.loop()，则是不停的调用MessageQueue的next()方法。
整个Handler机制中，核心的处理是在MessageQueue中。

MessageQueue

JDK注释

MessageQueue保存由Looper dispatch出来的Message。
Message不是直接add到MessageQueue中的，而是通过Looper绑定的Handler add进来的。
可以通过Looper.myQueue()得到当前线程的MessageQueue。

成员变量

  • boolean mQuitAllowed
    是否可终止，默认为true，由Looper的私有构造方法传入

    系统创建的UI线程的MessageQueue是不允许的，其他客户端代码创建的都是允许的

  • long mPtr
    Messqueue本身的native Messagequeue
  • Message mMessages
    消息链表，Message是个链表结构
  • ArrayList mIdleHandlers

  • IdleHandler[] mPendingIdleHandlers
    上面两个其实是同个东西，mPendingIdleHandlers是mIdleHandlers的数组版本。这两个成员都不是MessageQueue里的主流程，就先忽略。

  • boolean mQuitting
    是否是终止状态

  • boolean mBlocked
    是否为阻塞状态：需要等待pollonce()执行完
  • int mNextBarrierToken
    和IdleHandler有关，也不是MessageQueue的主流程。暂时忽略。

构造函数

MessageQueue的初始化。。是在native层完成的。。初始化了mPtr。

enqueueMessage()

step1,安全检查，如果Msg没有对应处理它的Handler或者Message已经在处理了，抛出异。
step2,如果MessageQueue终止了(可能是线程终止了)，则抛出异常并回收Message。
step3,正式处理Message之前，给Message设好使用中的标志位，以及处理它的时间。
step4，

step5,如果需要唤醒，则调用native方法唤醒。
小结：enqueueMessage干的是根据Message的执行时间，把Message插入到链表的合适位置。（链表本身是有序的，插入是个从头部开始的单向遍历过程）

next()

step1,安全检查。Looper调用了quit()之后是不允许再执行的，这种情况会直接return null。
step2,为第一次迭代设置必要的初始值。
step3,进入迭代。
step3.1,阻塞状态的话，释放被pending的对象。
step3.2,nativePollOnce()，其作用是如果阻塞的话一直等到取到Message。
step3.3,取出链表中第一条target不为null的消息。
step3.4,

step3.5，如果消息队列退出了，释放所有消息，return null。
step3.6，能运行到此处即代表消息队列处于空闲状态，空闲状态则初始化mPendingIdleHandlers。
step3.7，执行IdleHandler。
step3.8，进入下次迭代(可能有新消息进入队列)。
小结：如果有消息要执行，则取出一条消息交给Looper分发给Handler执行；如果没有消息要执行，则执行IdleHandler(如果有的话)。
