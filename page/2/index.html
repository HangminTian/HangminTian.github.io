Handler机制(3)

enqueueMessage、next

在Handler机制中，分析到Handler的enqueueMessage()方法，它最终的将msg交给了MessageQueue的enqueueMessage()处理。
而Looper.loop()，则是不停的调用MessageQueue的next()方法。
整个Handler机制中，核心的处理是在MessageQueue中。

MessageQueue

JDK注释

MessageQueue保存由Looper dispatch出来的Message。
Message不是直接add到MessageQueue中的，而是通过Looper绑定的Handler add进来的。
可以通过Looper.myQueue()得到当前线程的MessageQueue。

成员变量

  • boolean mQuitAllowed
    是否可终止，默认为true，由Looper的私有构造方法传入

    系统创建的UI线程的MessageQueue是不允许的，其他客户端代码创建的都是允许的

  • long mPtr
    Messqueue本身的native Messagequeue
  • Message mMessages
    消息链表，Message是个链表结构
  • ArrayList mIdleHandlers

  • IdleHandler[] mPendingIdleHandlers
    上面两个其实是同个东西，mPendingIdleHandlers是mIdleHandlers的数组版本。这两个成员都不是MessageQueue里的主流程，就先忽略。

  • boolean mQuitting
    是否是终止状态

  • boolean mBlocked
    是否为阻塞状态：需要等待pollonce()执行完
  • int mNextBarrierToken
    和IdleHandler有关，也不是MessageQueue的主流程。暂时忽略。

构造函数

MessageQueue的初始化。。是在native层完成的。。初始化了mPtr。

enqueueMessage()

step1,安全检查，如果Msg没有对应处理它的Handler或者Message已经在处理了，抛出异。
step2,如果MessageQueue终止了(可能是线程终止了)，则抛出异常并回收Message。
step3,正式处理Message之前，给Message设好使用中的标志位，以及处理它的时间。
step4，

step5,如果需要唤醒，则调用native方法唤醒。
小结：enqueueMessage干的是根据Message的执行时间，把Message插入到链表的合适位置。（链表本身是有序的，插入是个从头部开始的单向遍历过程）

next()

step1,安全检查。Looper调用了quit()之后是不允许再执行的，这种情况会直接return null。
step2,为第一次迭代设置必要的初始值。
step3,进入迭代。
step3.1,阻塞状态的话，释放被pending的对象。
step3.2,nativePollOnce()，其作用是如果阻塞的话一直等到取到Message。
step3.3,取出链表中第一条target不为null的消息。
step3.4,

step3.5，如果消息队列退出了，释放所有消息，return null。
step3.6，能运行到此处即代表消息队列处于空闲状态，空闲状态则初始化mPendingIdleHandlers。
step3.7，执行IdleHandler。
step3.8，进入下次迭代(可能有新消息进入队列)。
小结：如果有消息要执行，则取出一条消息交给Looper分发给Handler执行；如果没有消息要执行，则执行IdleHandler(如果有的话)。

本篇将从Handler相关的各个类源码来分析，解读Handler机制

Message

JDK注释

定义了Message类：带有描述和任意的数据对象，可以发送给Handler进行处理。
Message包含了2个额外的int成员和一个Object成员，从而允许使用的时候满足各种各样的case。
虽然Message的构造方法是public的，但是建议使用Handler.obtainMessage()和Message.obtain()，这样可以使Message循环使用。

成员变量

  • int what
    一个识别代码，用于接收消息时区分该消息，以确定应该做何处理。
  • Bundle data
    可以存一些复杂的数据，按我理解也是用来确定消息的。
  • int arg1, arg2
    arg1和arg2可以是data的一个替代，如果只需要存几个int值，what，arg1，arg2就够用了。
  • Object obj
    不能存在Bundle里的，可以存在Obj里。
  • Handler target
  • Runnable callback
  • Message next
    指向下个Message
  • Message sPool
    这是个static的成员， 指向的是当前Message

    小结

    从成员变量就可以看出，Message本身就是个单向链表结构，sPool指向当前节点，next指向下个节点。

构造方法

重要的api

  • obtain
    obtain有一系列的重载方法，其作用是从链表中取出一条Message（sPool），然后sPool = next，除了obtain(Message orig)这个方法是从orig中拷贝。

  • recycle()

其执行结果是把当前Message插入到链表的头部。如图：

  • sendToTarget
    这个方法非常短，它仅仅只是调用Handler把当前消息发出去了。

这个方法的存在，可以让我们免去 msg.getTarget().sendMessage(msg)这样的别扭操作。

MessageQueue

JDK注释

MessageQueue保存Looper"处理过"的Message。
这些Message不是直接添加进MessageQueue的，
而是通过Looper对应的Handler放进MessageQueue的。
可以调用Looper.myQueue()得到MessageQueue。

构造方法

MessageQueue这个类，连构方法都不是public的。其中nativeInit()是个本地方法

Looper

JDK注释

Looper是用于线程运行消息循环的类。
默认情况下线程没有与之对应的消息循环，
在线程中调用prepare()方法可以创建一个消息循环。然后调用loop()方法使之运行起来。
典型的使用例子如下：

成员变量

  • static final ThreadLocal sThreadLocal
  • static Looper sMainLooper
  • MessageQueue mQueue
  • Thread mThread

构造方法

构造方法中初始化了它的两个重要成员变量。值得注意的是这个构造方法是private的。

重要的api

从光Looper的注释看，prepare()和loop()这两个方法肯定是够重要的了。

  • prepare()

prepare()是Looper的初始化工作，它把Looper的实例传给ThreadLocal。
其中有个判断很有趣：

这个判断保证了每个线程都只能有一个Looper。
因为多次调用prepare()必会抛出RuntimeException。
假设如果没有这个判断，一个线程可以有多个Looper，
那么自然MessageQueue也会有多个，那么Handler机制就完全混乱了。

  • loop()

myLooper()如下

sThreadLocal.get()没空时，会抛出RuntimeException，这就解释了为什么必须先调用Looper.prepare()。
接下去是一个死循环，
1.挨个取链表中的Message，交给Handler进行diapatchMessage(msg)，
2.将Message从链表中抹去。
3.直到所有Message都dispatch完，return。

提前看一下dispatchMessage()方法。

1.如果Message本身的Runnable不为null，则执行Message本身的Runnable的run()方法；
2.Handler的Runnable不为null， 则执行Handler本身的Runnable的run()方法；
3.前2者都不满足，执行handleMessage()方法。
这里解释了为什么post()、sendMessage()都可以在子线程中完成UI操作。

Handler

JDK注释

Handler可以发送消息和Runnable对象到MessageQueue。
Handler、线程、MessageQueue是一一对应的。
每个创建出来的Handler实例都会绑定线程的MessageQueue，
Handler传递Message和Runnable，而负责MessageQueue执行。
Handler的两大用途是：
1、延时执行Message和Runnable
2、在另一个线程中执行Message和Runnable

成员变量

  • MessageQueue mQueue
  • Looper mLooper
  • Callback mCallback
  • boolean mAsynchronous
  • IMessenger mMessenger

可以发送消息的API

  • post (调用的sendMessageDelayed)
  • postAtTime (调用的sendMessageAtTime)
  • postDelayed (调用的sendMessageDelayed)
  • sendMessageAtTime
  • sendMessageDelayed （调用的是SendMessageAtTime）
  • sendMessage （调用的sendMessageDelayed）
  • sendEmptyMessageDelayed (调用的sendMessageDelayed)
  • sendEmptyMessage （调用的sendEmptyMessageDelayed）

这么多API，最终调用的都是sendMessageAtTime。

最终调用的enqueueMessage()。

enqueueMessage

1.设置Message的Target为this，即当前Handler是这条消息的处理者；
2.设置异步flag
3.将消息放到消息队列里
